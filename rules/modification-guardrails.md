# 수정 가드레일 (Modification Guardrails)

AI Agent가 코드 및 개발 인프라를 수정할 때 반드시 준수해야 하는 범위·강도·프로세스 규칙.

> §1~2 (패키지 경계, 도메인 식별)는 프로젝트별 configurable rule (`domain-boundaries.md`)에서 정의.
> 본 파일은 **모든 프로젝트에 공통**인 §3~6을 담습니다.

## 3. 최소 수정 원칙

IMPORTANT: 모든 코드 수정에 적용됩니다.

1. 요청된 기능/버그에 **직접 관련된 파일만** 수정
2. 범위 외 파일 수정이 필요하면 **즉시 멈추고 사용자에게 확인**
3. 리팩토링, 코드 정리 등 부수적 변경은 별도 요청 없이 수행하지 않음

## 4. Anti-Overengineering

1. **불필요한 추상화 금지**: "나중에 필요할 수 있으니" 인터페이스·팩토리·제네릭을 만들지 않는다
2. **선제적 최적화 금지**: 성능 문제가 측정되지 않았으면 최적화하지 않는다
3. **의존성 추가 최소화**: 10줄 이내 코드로 해결 가능하면 패키지를 추가하지 않는다
4. **워크플로우 과잉 금지**: Small 작업에 Large 프로세스를 적용하지 않는다. 모호하면 작은 쪽으로 판단

### 자가 점검 (구현 완료 시)

매 구현 완료 시 점검:

- 사용자가 요청하지 않은 파일을 수정했는가?
- "확장성을 위해" 추가한 코드가 있는가?
- 제거해도 요청된 기능이 동작하는 코드가 있는가?

→ 하나라도 YES이면 해당 코드를 제거한다.

## 5. 도구/인프라 변경 프로토콜 (Harness Change Protocol)

IMPORTANT: `.claude/`(Hook, Rule, Skill, Agent) 또는 `scripts/` 변경 시 반드시 적용.

> **핵심 원칙: "Script가 할 수 있는 것은 Script에, LLM만 할 수 있는 것은 LLM에"**

### 5.1 변경 전 3-질문 필터

| #             | 질문                                                   | YES이면                             |
| ------------- | ------------------------------------------------------ | ----------------------------------- |
| **Q1** 충돌   | 기존 Hook/Rule/Skill/빌트인 명령과 역할이 겹치는가?    | **제안 중단** — 기존 구조 확장 검토 |
| **Q2** 실효성 | 이론적 개선이 아닌, 실제 워크플로우에서 체감 가능한가? | NO이면 **제안 중단**                |
| **Q3** 불필요 | 이미 잘 동작하는 것을 굳이 바꾸는 건 아닌가?           | YES이면 **제안 중단**               |

### 5.2 "기존 확장 우선" 원칙

**확인 순서:**

0. **모델 능력 변화 검토** — 현재 작성하려는 가드/스크립트가 향상된 모델 성능 덕분에 더 이상 필요 없는지 자문
1. **Claude Code 최신 네이티브 기능** — 정적 지식에 의존하지 않고 최신 기능 조회
2. **시스템 프롬프트 자동 로드** — `.claude/rules/*.md`는 이미 0T 추가 비용으로 context에 존재
3. **기존 Hook** — `.claude/hooks/*.sh`의 역할과 중복 여부
4. **기존 Skill** — `.claude/skills/`의 워크플로우와 중복 여부
5. **기존 Agent** — `planning-agent`, `review-agent`의 담당 범위와 중복 여부

### 5.3 해결 수단 우선순위

```
1. 기존 문서 수정 (Rule/Skill 1~3줄 추가)     ← 가장 선호
2. 기존 Hook 로직 확장 (조건 추가, 출력 확장)
3. 기존 스크립트에 기능 추가
4. 신규 스크립트 작성                           ← 가장 후순위
```

### 5.4 Harness 변경 자가 점검

Harness 변경 완료 시 점검:

**중복 방지**:

- 기존 Hook/Rule/Skill과 역할이 중복되는 파일을 새로 만들었는가?
- LLM-only 영역(의미 판단, 자연어 해석)을 스크립트로 자동화하려 했는가?
- **Agent 컨텍스트 예외**: Agent/Sub-Agent는 rules를 상속받지 않으므로, 핵심 행동 규칙은 agent 파일에 요약본 기재 허용

**실행 검증 (신규 또는 수정된 Hook/Script 전용)**:

1. **Exit 코드 검증**: `echo '{}' | bash <hook> 2>&1; echo "EXIT=$?"` → EXIT=0 확인
2. **출력 스모크 테스트**: 조건부 출력이면 조건을 만족하는 상태로 1회 확인
3. **Bash 방어 패턴**: `set -euo pipefail` 환경에서 `|| true` 가드 필수:
   - `VAR=$(cmd | grep ... | head -N)` — SIGPIPE 발생 가능
   - `if cmd | grep -q ...; then` 형태는 가드 불필요
4. **Hook 전수 회귀 테스트**: `for h in .claude/hooks/*.sh; do echo '{}' | bash "$h" >/dev/null 2>&1 && echo "$h OK" || echo "$h FAIL"; done`
