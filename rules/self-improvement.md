# Self-Improvement Rule

## 교훈 기록

- 사용자가 코드나 접근 방식을 **수정/지적**하면 → `docs/lessons.md`에 기록
- 사용자가 워크플로우(스킬/규칙/Agent/문서 구조)를 **수정/지적**하면 → 동일하게 기록
- 기록 형식: `## [category: <카테고리명>] <교훈 제목>` + 날짜, 상황, 실수, 교훈
- 카테고리 예시:
  - 코드: `validation`, `tdd`, `architecture`, `security`, `naming`, `scope`, `communication`
  - 워크플로우: `workflow`, `skill`, `rule`, `agent`
  - 인프라: `infra`, `tooling`

### 스킬 실행 중 교훈의 배치

- 사용자가 **특정 스킬 실행 중** 접근 방식을 수정/지적하면:
  - 해당 스킬에 `## Pitfalls` 섹션이 있으면 → 거기에 1줄 추가 + lessons.md에도 기록
  - Pitfalls 섹션이 없으면 → lessons.md에만 기록 (기존 동작)
- Pitfalls 항목이 5건 이상이면 → `references/` 파일로 분리 제안 (Progressive Disclosure)

## 반복 요청 기록

- 동일 유형의 작업 요청을 **2회 이상** 수행했다고 인지하면 → `docs/lessons.md`에 `[request: <요청유형>]` 형식으로 기록
- "동일 유형" 판단: 작업의 목적과 대상이 같은 경우
- 기록 시점: 해당 작업 완료 직후 (세션 중)
- 카테고리는 구체적으로 명명. 과도하게 넓은 카테고리 금지
- **즉시 제안 의무**: `[category: X]` 또는 `[request: X]` 항목을 lessons.md에 기록할 때, 해당 카테고리/요청 유형의 **기존 항목 수를 확인**한다. 이번 기록으로 3건 이상이 되면, 기록 직후 사용자에게 승격(category) 또는 자동화(request) 승인 여부를 즉시 질문한다

## 규칙 승격 (로그 기반 자동 감지)

- `session-start-validator.sh`가 세션 시작 시 `docs/lessons.md`를 파싱
- 동일 카테고리 항목이 **3건 이상**이면 → 승격 후보로 사용자에게 제안
- **안전망 행동 의무**: Hook 출력에 `ACTION REQUIRED`가 포함된 경우(= 이전 세션에서 즉시 제안이 누락된 후보), 사용자의 첫 번째 메시지 응답 시 승격/자동화 승인 여부를 **반드시 질문**한다
- 사용자 확인 후 → 이 파일 "승격된 규칙" 섹션에 추가 + 해당 항목을 `docs/lessons-archive.md`로 이동
- **아카이브**: 승격 완료 항목 및 6개월간 동일 카테고리 재발 없는 항목은 `docs/lessons-archive.md`로 이동
- **상한**: lessons.md 항목이 30건을 초과하면 가장 오래된 아카이브 대상부터 이동
- `[request: X]` 패턴도 동일하게 3건 이상이면 → 자동화 후보로 사용자에게 제안 (**반드시 사용자 확인 후 진행**)
- review-agent도 코드 리뷰 중 반복 패턴을 독립적으로 보고 (정성적 보완)
- **승격 형태 결정 기준**: 실수의 성격에 따라 가장 적합한 형태를 선택한다 (Harness Change Protocol §6 준수)
  - **Hook 확장**: 패턴 매칭으로 결정론적 검출이 가능한 실수
  - **ESLint 규칙 추가**: 코드 문법·타입 수준에서 린트로 강제 가능한 실수
  - **Skill 생성/수정**: 반복 워크플로우를 자동화하면 해소되는 실수
  - **Rule 파일 추가**: 의미 판단·맥락 이해가 필요한 실수
  - **Agent 수정**: Agent 핸드오프·생명주기에서 발생하는 실수
  - 결정 원칙: 구조적 강제(Hook/ESLint)가 가능하면 우선 검토하되, **실수의 본질에 맞는 형태**를 선택한다
- **구조적 편향 승격**: AI 구조적 편향에서 비롯된 이슈가 반복되면 → `self-improvement.md` 구조적 편향 방어에 원칙 추가

## 구조적 편향 방어

코드 작성·수정 시 아래를 준수한다.

**변경 후 역추적**: 변경된 변수·함수를 grep하여 미사용 잔재(데드 변수, 옛 주석) 정리. 변수명이 실제 역할과 일치하는지 확인.

**동일 패턴 전파**: 패턴(regex, 상수, 시그니처) 변경 시 grep으로 동일 패턴 사용처를 전수 검색하여 일괄 변경. Hook/컴포넌트 추가 시 실행 체인 내 역할 중복 확인.

**완전성 검증**: 규칙·상태값 도입 시 CRUD 전체 생명주기의 **실행 배선**을 확인. "존재하는가"가 아니라 "누가·언제 실행하는가"까지 검증. 파일 append 시 로테이션/상한 필수. 비정상 종료 시나리오 고려.

**실행 성공 검증 (침묵 실패 방지)**: 스크립트·Hook·빌드 명령을 추가·수정하면 **실제 실행 → exit 0 확인 → 예상 출력 확인까지가 한 동작이다**. "작성했다"는 "작동한다"가 아니다. 구체적 체크리스트는 `modification-guardrails.md` §6.4 "실행 검증" 참조.

**도달 경로 검증**: 규칙·점검·방어 장치를 추가할 때, 해당 파일이 의도한 실행 시점에 실제로 로드되는지 **추가 즉시** 확인. paths 조건부 로딩 구조에서는 대상 파일의 paths가 실제 편집 경로를 포함하는지 확인.

**비정상 경로 시뮬레이션**: 데이터 보호·백업·미러링 메커니즘을 설계할 때, "원본이 손상된 상태에서 이 메커니즘이 실행되면?"을 반드시 시뮬레이션.

## 폐기 및 진화 검토

### 사용 빈도 기반

- 승격 규칙 중 6개월간 관련 paths 파일 변경 없는 항목은 제거 후보로 사용자에게 제안
- `/harness-evaluation` 수행 시, 6개월간 호출 기록 없는 스킬을 폐기 후보로 보고

### 외부 변화 기반 (자가-진화 루프)

- 마지막 외부 변화 점검(`dev/docs/harness-evolution/last-scan.txt`)으로부터 30일 경과 시 → `session-start-validator.sh`가 첫 사용자 메시지에 1회 알림
- 사용자가 `/harness-evaluation`을 호출하면 분석 후 자가-진화 8단계 루프 진입
- 사용자가 거부/연기한 후보는 `dev/docs/harness-evolution/decisions.md`에 기록 (30일 쿨다운 후 재제안)

### 단순화 기반 (모델 능력 향상)

- `/harness-evaluation` 10번 축에서 단순화 후보가 발견되면 동일 절차 적용
- 단, 결정론적 보안/빌드 가드는 단순화 대상에서 영구 제외

## 자가-진화 루프 (Self-Evolution Loop)

외부 변화(Claude Code 네이티브 기능 출시 또는 모델 성능 향상)에 대응하여 하네스를 진화시키는 8단계 절차.
**모든 단계는 결정 분기점**이므로 `decision-autopilot.md`가 자동 적용된다.

| 단계         | 동작                                                           | 책임 자산                  |
| ------------ | -------------------------------------------------------------- | -------------------------- |
| 1. 탐색      | 마지막 점검일 후 30일 경과 시 알림                             | session-start-validator.sh |
| 2. 발견      | WebSearch로 Claude Code changelog + 모델 릴리스 조회           | 사용자가 명시 호출         |
| 3. 분석      | harness-evaluation 8번 + 10번 축만 부분 호출                   | harness-evaluation 스킬    |
| 4. 비교검토  | 후보별 표 작성 (자산 / 현재 역할 / 대체안 / 위험 / 권장)       | 일반 P4 작업               |
| 5. 기획/설계 | 결정 자동조종 자동 적용 (듀얼 추천 + 의도 탐지)                | decision-autopilot.md      |
| 6. 제안      | AskUserQuestion 4-옵션 (Shadow 채택 / 즉시 채택 / 거부 / 연기) | 결정 자동조종              |
| 7. 실행/구현 | 일반 P4 + Shadow 미러 생성 (`.claude/deprecated/`)             | modification-guardrails §6 |
| 8. 검증      | Shadow 14일 후 만료 알림 → 사용자가 최종 삭제 또는 롤백        | session-start-validator.sh |

**원칙**:

- 모든 단계의 마지막은 사용자 승인 (자동 실행 금지)
- 결정론적 보안/빌드 가드는 단순화 대상 영구 제외
- 사용자 거부/연기는 `dev/docs/harness-evolution/decisions.md`에 누적, 30일 쿨다운 후 재제안 가능

**Shadow 미러 명명 규칙**:

- 원본: `.claude/skills/foo/SKILL.md`
- Shadow: `.claude/deprecated/skills_foo_SKILL.md.shadow-expires-YYYY-MM-DD`
- 만료일은 ISO 8601, 기본 14일
