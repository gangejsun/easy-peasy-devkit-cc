# Harness Evaluation

Harness 구성 요소 전체를 10축 체계로 정량+정성 평가하고, 이전 평가 대비 변화를 추적한다.

## 워크플로우

### Step 1: 평가 범위 결정

| 조건                           | 동작                                 |
| ------------------------------ | ------------------------------------ |
| 인자 없음 (기본)               | 전체 10축 평가 → Step 2→3→4→5        |
| 특정 축 지정 (예: "보안 축만") | 해당 축만 심층 평가 → Step 2→4→5     |
| 이전 버전 비교 요청            | 전체 10축 + diff 강조 → Step 2→3→4→5 |

### Step 2: Harness 현황 수집

아래 경로를 탐색하여 현재 구성 요소를 수집한다:

| 구성 요소        | 탐색 경로                                                                                                 | 수집 항목                                                                              |
| ---------------- | --------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Rules            | `.claude/rules/*.md`                                                                                      | 파일 수, paths 지정 여부, 핵심 규칙 목록                                               |
| Hooks            | `.claude/hooks/*.sh` + `settings.json` hooks 섹션                                                         | Hook 수, 이벤트 타입별 분류, 실행 조건                                                 |
| Skills           | `.claude/skills/*/SKILL.md`                                                                               | 스킬 수, 각 스킬 description, 하위 폴더 구조                                           |
| Agents           | `.claude/agents/*.md`                                                                                     | Agent 수, 담당 Phase, handoff 규칙                                                     |
| Scripts          | `scripts/*.sh`                                                                                            | 스크립트 수, 각 역할                                                                   |
| CLAUDE.md        | `CLAUDE.md`                                                                                               | 행 수, Phase 테이블, 워크플로우 구조                                                   |
| 교훈             | `docs/lessons.md`                                                                                         | 항목 수, 카테고리 분포, 승격 이력                                                      |
| 테스트/CI/Git 훅 | `package.json`, `vitest.config.*`/`jest.config.*`, `.github/workflows/`, `.husky/`, `commitlint.config.*` | test 스크립트 유무, 테스트 파일 수, CI 파이프라인 유무, Git 훅 유무, PreCommit 훅 유무 |

### Step 2.5: 설계 히스토리 검증 (이전 평가 판정 계승 방지)

IMPORTANT: 이전 평가에서 "미해결", "중복", "안티패턴"으로 판정된 항목이 있으면, 해당 판정을 **그대로 계승하지 않고** 아래를 반드시 확인한다:

1. **"paths 미지정 = 안티패턴" 판정 검증**: 해당 파일이 매 세션 로드가 의도된 라우터인지 확인
2. **"중복" 판정 검증**: 유사한 파일이 2개 이상 존재할 때, 설정 기반 선택 구조로 연동되는지 확인
3. **"미해결" 판정 검증**: 이전 평가의 "개선 방향"이 다른 형태로 이미 해결되었을 수 있음

### Step 3: 10축 평가 수행

각 축을 0~5 (0.5 단위)로 평가한다. 평가 시 **구체적 근거**(파일명, 규칙, 메커니즘)를 반드시 명시한다.

**10축 정의:**

| #   | 축                 | 평가 대상                                                                                    | 핵심 질문                                                            |
| --- | ------------------ | -------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| 1   | 기획 파이프라인    | P0~P3 Phase 체계                                                                             | 아이디어→PRD→개발 문서까지 빈틈 없는가?                              |
| 2   | 토큰 효율          | 3-Layer 모델, paths 지정, 조건부 로딩                                                        | 불필요한 컨텍스트 소비가 없는가?                                     |
| 3   | 확인 편향 방지     | 구현/검증 분리, review-agent 독립성                                                          | 자기 코드를 자기가 검증하는 구조인가?                                |
| 4   | 컨텍스트 연속성    | pre-compact-saver, dev/active/, Persistent Loop                                              | 세션 전환 시 정보 손실이 있는가?                                     |
| 5   | 점진적 거버넌스    | Small/Medium/Large 차등 적용                                                                 | 작은 작업에 과도한 오버헤드가 있는가?                                |
| 6   | 자기 개선          | lessons.md, 규칙 승격, 구조적 편향 방어, 승격 규칙 도달 경로                                 | 반복 실수를 자동으로 학습하고, 승격된 규칙이 실제로 적용되는가?      |
| 7   | 보안               | Rule/Hook/Skill 3중 방어, 비밀키 차단, Git 훅 방어선                                         | 보안 사고를 구조적으로 예방하는가?                                   |
| 8   | 도구 생태계        | 스킬 다양성, Plugin 시스템, 마케팅 파이프라인, Claude Code 네이티브 vs 커스텀 비교           | 네이티브로 충분 대체 가능한 커스텀 자산이 남아있는가?                |
| 9   | SW 엔지니어링 기반 | 테스트 인프라, CI/CD 파이프라인, Git 훅, PreCommit                                           | 하네스가 보호하는 코드 자체가 검증 가능한 상태인가?                  |
| 10  | 모델 능력 정합성   | Hook/Rule이 현재 모델 성능 대비 과잉인지 점검                                                | 모델이 직접 안전하게 처리 가능한 작업을 Script로 잠그고 있지 않은가? |

**평가 기준 (각 축 공통):**

| 별점 | 기준                             |
| ---- | -------------------------------- |
| 5    | 업계 최고 수준. 구조적 결함 없음 |
| 4    | 우수. 미세 개선 여지만 존재      |
| 3    | 양호. 명확한 개선 포인트 1~2개   |
| 2    | 기초 수준. 주요 Gap 존재         |
| 1    | 미흡. 근본적 재설계 필요         |
| 0    | 해당 기능 부재                   |

### Step 4: 강점/한계 분석

- **강점**: 평가 중 발견한 구조적 우수점 (최대 10개, 영역+설명 테이블)
- **한계 및 개선 여지**: 현재 한계 + 구체적 개선 방향 (최대 5개, 영역+한계+개선 방향 테이블)
- 이전 평가 결과가 있으면 **변화 추적**: 축별 점수 변동 + 변동 사유

### Step 5: 보고서 생성 및 기록

1. 보고서 형식: `assets/report-template.md` 참조
2. 이전 평가 버전 확인: `docs/lessons.md`에서 `[request: harness-evaluation]` 항목의 최신 버전 번호를 읽고, +1 하여 새 버전 부여
3. **보고서 저장**: `dev/docs/harness-evaluation/v{N}.md`에 저장
4. **lessons.md 기록**: 평가 완료 후 `[request: harness-evaluation]` 항목 추가
5. **README.md 업데이트**: 강점/한계 테이블을 최신 평가 결과로 갱신

## 주의사항

- 평가는 **현재 파일 상태** 기준이되, 이전 평가 판정을 검증 없이 계승하지 않는다 (Step 2.5 필수)
- 각 축 평가 시 "구체적 파일/메커니즘 → 점수"의 근거 체인을 유지한다
- 이전 평가와 점수가 동일해도 근거가 다르면 명시한다
- 종합 점수는 10축 산술 평균 (소수점 첫째 자리까지)
