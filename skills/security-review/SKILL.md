# Security Review

OWASP Top 10 기반 보안 취약점 검사 및 구조화된 보고서 생성.

## 워크플로우

### Step 1: 검토 범위 결정

| 검토 유형 | 트리거 | 검사 범위 | 실행 Step |
|----------|--------|----------|----------|
| 전체 검토 | 사용자가 전체 보안 감사 요청 | 프로젝트 전체 | 2 → 3 → 4 → 5 → 6 |
| 변경분 검토 | 기능 구현 완료 후, PR 리뷰 | `git diff` 대상 파일 | 2 → 3 → 4 → 6 |
| 특정 영역 검토 | 특정 도메인/기능 보안 확인 | 지정된 경로만 | 2 → 3 → 6 |

### Step 2: 시크릿 및 환경변수 검사

검사 항목:

| 패턴 | 심각도 | 검사 방법 |
|------|--------|----------|
| 하드코딩된 API 키/토큰 | Critical | grep으로 `sk_`, `pk_`, `key=`, `token=`, `password=` 등 탐지 |
| 환경파일 git 추적 | Critical | `.gitignore`에 `.env*` 포함 여부 |
| 클라이언트에 서버 전용 시크릿 노출 | Critical | 클라이언트 코드에서 서버 전용 키 사용 여부 |
| 환경변수 접두사 남용 | High | 서버 전용 값에 클라이언트 접두사 사용 여부 |

### Step 3: 입력 검증 및 인증/인가 검사

**입력 검증:**

| 대상 | 검사 내용 | 심각도 (미준수 시) |
|------|----------|-----------------|
| API 엔드포인트 | 스키마 기반 입력 검증 | High |
| 서버사이드 액션 | 인자 검증 | High |
| 동적 라우트 파라미터 | 파라미터 타입 검증 | Medium |
| 파일 업로드 | 타입/크기 제한 | Medium |

**인증/인가:**

| 대상 | 검사 내용 | 심각도 (미준수 시) |
|------|----------|-----------------|
| 서버사이드 액션 | 인증된 사용자 확인 | Critical |
| API 엔드포인트 | 세션/토큰 검증 | Critical |
| 미들웨어/가드 | 보호 경로 정의 + 세션 검증 | High |
| DB 접근 제어 | 행 수준 보안(RLS) 또는 쿼리 필터 | Critical |

### Step 4: 인프라 보안 검사

CLAUDE.md의 패키지 매니저를 확인하고 의존성 감사를 수행:

```bash
# 프로젝트 패키지 매니저에 맞는 감사 명령 실행
# npm audit / pnpm audit / yarn audit / pip-audit / cargo audit 등
```

| 검사 항목 | 심각도 |
|----------|--------|
| Critical/High CVE 의존성 | Critical/High |
| 보안 헤더 미설정 (CSP, X-Frame-Options 등) | Medium |
| 안전하지 않은 HTML 삽입 | High |
| 에러 응답에 내부 정보 노출 | Medium |

### Step 5: 결제/민감 데이터 보안 검사

결제 또는 민감 데이터 처리 코드가 존재하는 경우에만 수행:

| 검사 항목 | 심각도 (미준수 시) |
|----------|-----------------|
| 금액/수량 서버사이드 검증 (클라이언트 전송값 불신) | Critical |
| 웹훅/콜백 시그니처 검증 | Critical |
| 시크릿 키 서버 사이드 전용 사용 | Critical |
| 상태 변경 서버사이드 처리 | High |

### Step 6: 보안 보고서 생성

다음 형식으로 보고서를 작성:

```
보안 검토 보고서
================

검토 범위: [전체/변경분/특정 영역] ([N]개 파일)
검토 일시: [YYYY-MM-DD]

CRITICAL (N건)
--------------
1. [파일:행] - [취약점 제목]
   발견: [구체적 설명]
   영향: [악용 시 영향]
   수정: [구체적 수정 방법]
   참조: OWASP [항목]

HIGH (N건)
----------
...

MEDIUM (N건)
------------
...

LOW (N건)
---------
...

의존성 취약점
-------------
[패키지 매니저 audit 결과 요약]

종합 평가
---------
보안 수준: [양호/주의/위험/심각]
즉시 조치 필요: [목록]
권고사항: [목록]
```

## 심각도 정의

| 심각도 | 기준 | 수정 기한 |
|--------|------|----------|
| **Critical** | 데이터 유출, 인증 우회, 시크릿 노출, RCE | 즉시 |
| **High** | 권한 상승, XSS, 입력 검증 부재 | 24시간 내 |
| **Medium** | 정보 노출, 보안 헤더 미설정, 검증 미흡 | 다음 릴리스 전 |
| **Low** | 모범 사례 미준수, 개선 권고 | 백로그 |

## 보안 체크리스트

### 인증/인가
- [ ] 서버사이드 액션에서 인증 사용자 확인
- [ ] API 엔드포인트에서 세션/토큰 검증
- [ ] 미들웨어/가드에서 보호 경로 정의
- [ ] DB 접근 제어 정책 활성화 (RLS 또는 쿼리 필터)
- [ ] 서버 전용 키가 서버에서만 사용

### 입력 검증
- [ ] 모든 API 엔드포인트 입력에 스키마 검증 적용
- [ ] 모든 서버사이드 액션 입력에 스키마 검증 적용
- [ ] URL 파라미터/쿼리 검증
- [ ] 파일 업로드 타입/크기 제한

### 시크릿 관리
- [ ] API 키/비밀번호 하드코딩 없음
- [ ] 환경파일이 `.gitignore`에 포함
- [ ] 클라이언트에 서버 전용 값 미노출

### 출력 보안
- [ ] 안전하지 않은 HTML 삽입 미사용 (또는 sanitize)
- [ ] 에러 응답에 내부 정보 미노출
- [ ] API 응답에 불필요한 필드 미포함

### 의존성
- [ ] 패키지 매니저 audit에서 Critical/High 없음
- [ ] 알려진 취약점 패키지 미사용
