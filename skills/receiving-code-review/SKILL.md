# Receiving Code Review

코드 리뷰 피드백을 기술적으로 검증하고, 심각도별로 적절히 처리한다.

## 핵심 원칙

- **검증 우선**: 피드백을 맹목적으로 수용하지 않는다. 실제 코드를 확인하여 정확성을 검증한다.
- **기술적 엄밀성**: 사회적 동의("좋은 지적이네요!")보다 기술적 판단을 우선한다.
- **근거 기반 반론**: 리뷰어가 틀렸을 때 구체적 근거와 함께 반박한다.
- **YAGNI 검증**: "전문적으로 보이는" 제안이 실제로 필요한지 검증한다.

## 금지 응답 패턴

피드백 수신 시 다음 표현을 사용하지 않는다:

- "좋은 지적입니다!" / "훌륭한 피드백입니다!" (수행적 동의)
- "바로 수정하겠습니다" (검증 전 즉시 수용)
- 장황한 사과나 과도한 설명

대신 사용할 표현:

- "수정 완료. [변경 내용 요약]" (수용 시)
- "[근거]에 의해 현재 구현이 적절합니다." (기각 시)
- "확인 결과 맞습니다. [이유]. 수정합니다." (반론 후 수용 시)

## 워크플로우

### Step 1: 피드백 소스 확인 및 처리 범위 결정

| 소스 | 실행 경로 |
|------|----------|
| /requesting-code-review 결과 (P5 내부) | Step 2 → Step 3 → Step 4 |
| P6 (gemini-claude-loop) 결과 | Step 2 → Step 3 → Step 4 |
| 사용자가 전달한 외부 리뷰 피드백 | Step 2(소스 신뢰도 낮음으로 검증 강화) → Step 3 → Step 4 |

### Step 2: 피드백 검증

각 이슈에 대해 다음을 확인한다:

| 검증 항목 | 방법 | 결과 |
|----------|------|------|
| 사실 확인 | 실제 파일:행을 열어 코드 확인 | 오탐(false positive) 걸러냄 |
| 심각도 적절성 | 프로젝트 맥락에서 영향 재평가 | 필요시 심각도 조정 |
| YAGNI 검증 | "이걸 도입하면 해결되는 현재 문제가 있는가?" | 불필요한 제안 격하/기각 |
| 의도적 결정 확인 | plan.md, context.md에서 사전 결정 여부 확인 | 의도적이면 기각 사유 기록 |
| 기존 기능 영향 | 제안 수용 시 기존 기능이 깨지는가 | 부작용 있으면 기각/보류 |

YAGNI 검증 상세:

```
제안: "[기능/패턴] 도입이 필요합니다"
확인:
1. 현재 코드베이스에서 실제 사용처가 있는가? (grep 확인)
2. 현재 규모에서 이 추상화가 정당한가?
3. 해결하려는 문제가 실제로 존재하는가?
→ 불필요하면 Suggestion으로 격하 또는 기각
```

검증 결과 형식:

```
피드백 검증 결과:
- 수용: [N]건 (원본 심각도 유지)
- 심각도 조정: [N]건 ([원본→조정] 사유 포함)
- 기각: [N]건 (근거 포함)
```

### Step 3: 심각도별 처리

**처리 순서**: Critical (blocking) → Important (단순→복잡) → Suggestion (기록만)

| 심각도 | 처리 | 시점 |
|--------|------|------|
| Critical | 즉시 사용자 보고. P4 재진입 필요 여부 확인. | P5 중단 |
| Important | 사용자 보고 후 수정 진행. 수정 완료 시 P5 계속. | P5 내 수정 |
| Suggestion | 보고서에 기록. 별도 수정 불필요. | 기록만 |

**Critical 발견 시 보고 형식**:

```
[Critical 이슈 발견]

1. [파일:행] - [이슈 설명]
   영향: [영향 범위]
   수정 방향: [권고 방법]

P4로 돌아가 수정을 진행할까요?
```

**Important 처리 분기**:
- 단순 수정 (1~2행 변경, 명확한 수정 방향): 직접 수정 후 보고
- 복잡한 수정 (로직 변경, 설계 결정 필요): 사용자에게 보고 후 방향 결정

**불명확한 피드백 처리**:

피드백 항목이 불명확하면 부분 처리하지 않고 전체 명확화를 요청한다. 항목 간 연관성이 있을 수 있으므로 부분 이해로 잘못된 수정을 하는 것을 방지한다.

### Step 4: 처리 결과 요약

```
피드백 처리 완료:
- Critical: [N]건 → [처리됨/P4 재진입 필요]
- Important: [N]건 → [처리됨/사용자 확인 대기]
- Suggestion: [N]건 → 기록됨
- 기각: [N]건 → [근거 요약]

[Critical/Important가 모두 처리된 경우]
→ 문서 최신화(completion-review)로 진행합니다.
```

## 반론 작성 가이드

리뷰어의 지적이 부정확할 때:

```
이 지적에 대해 다른 견해가 있습니다:

지적: [리뷰어의 지적 요약]
반론: [구체적 근거]
참조: [관련 코드/문서/공식 문서 경로]
결론: [수용/기각/부분 수용] + 사유
```

반론이 타당한 상황:
- 제안이 기존 기능을 깨뜨리는 경우
- 리뷰어가 전체 컨텍스트를 파악하지 못한 경우
- YAGNI 위반 (사용되지 않는 기능/패턴 도입)
- 프로젝트 기술 스택에서 기술적으로 부정확한 경우
- 사용자의 아키텍처 결정과 충돌하는 경우

반론이 틀렸을 때:

```
확인 결과 [지적 내용]이 맞습니다. [검증 근거]. 수정합니다.
```
