# Requesting Code Review

git diff 기반으로 구현 코드를 6개 영역에서 체계적으로 리뷰하고 구조화된 보고서를 생성한다.

## 워크플로우

### Step 0: 리뷰 모드 자동 선택

git diff 분석 후 작업 규모를 자동 분류하여 리뷰 모드를 결정한다.

| 작업 규모 | 리뷰 모드 | 검증 범위 | 예상 시간 |
|----------|----------|----------|----------|
| Medium | skip | 검증 생략 (build/test만) | 0분 |
| Medium | minimal | Critical만 (타입, 도메인, 보안, 에러) | 3~5분 |
| Large | full | 6-area 전체 검증 | 10~15분 |

#### Medium 자동 감지 로직

다음 조건을 **모두** 만족하면 Medium로 분류하여 **리뷰를 즉시 종료**한다:

**Medium 조건 (AND)**:
1. 변경 파일 1개
2. 변경 줄 수 ≤ 10줄
3. 다음 중 하나:
   - 스타일/클래스명만 변경
   - 하드코딩된 문자열만 변경 (변수명/함수명 제외)
   - import 추가/제거만 (코드 로직 변경 없음)

**Medium 제외 (OR - 하나라도 해당하면 Medium 이상)**:
- API 라우트/핸들러 파일
- 서버 액션 파일
- 미들웨어 파일
- 테스트 코드
- DB 관련 파일 (마이그레이션, 스키마 등)
- 타입 정의 파일
- 함수 시그니처 변경
- 조건문(if/switch) 추가/수정
- 상태 관리 로직 추가/수정

**Medium 감지 시 출력**:
```
코드 리뷰 생략 (Trivial)
======================

작업 규모: Trivial (1파일, N줄, 스타일만)
검증: build/test로 충분
결과: ✅ 승인 (P5 Step C로 진행)
```

### Step 1: 리뷰 컨텍스트 수집 및 분기 결정

**이 Step은 Medium(minimal 모드) 또는 Large(full 모드)에서만 실행됩니다.**

리뷰에 필요한 정보를 수집하고, dev/active/ 존재 여부에 따라 이후 실행 경로를 결정한다.

| 조건 | 실행 경로 |
|------|----------|
| dev/active/<name>/ 존재 (계획 기반 구현) | Step 2 → Step 3 → Step 4 → Step 5 |
| dev/active/ 부재 (버그 수정 등) | Step 3 → Step 4 → Step 5 |

수집할 정보:

```bash
# 변경 파일 목록
git diff --stat HEAD~1

# 전체 변경 내용
git diff HEAD~1

# dev/active 존재 확인
ls dev/active/<name>/ 2>/dev/null
```

### Step 2: 계획 정합성 검증

dev/active/<name>/ 문서 대비 구현을 검증한다. dev/active/가 없으면 이 Step을 건너뛴다.

| 검증 항목 | 확인 내용 |
|----------|----------|
| tasks.md 커버리지 | 모든 task 항목이 구현되었는가 |
| plan.md 방향성 | 계획된 아키텍처/접근법대로 구현되었는가 |
| Scope 준수 | 수정 범위를 벗어나지 않았는가 |
| 미구현 항목 | 의도적 제외인지, 누락인지 판단 |

검증 결과 형식:

```
계획 정합성:
- tasks.md 커버리지: [N]/[M] 항목 구현
- Scope 준수: [준수/위반 - 상세]
- 편차: [없음 / 편차 목록 + 정당성 평가]
```

### Step 2-1: PRD 요구사항 대조 (dev/docs/prd/ 존재 시)

dev/docs/prd/ 에 해당 기능의 PRD가 존재하면, PRD의 기능 요구사항을 하나씩 나열하고 구현 상태를 대조한다. PRD가 없으면 이 Step을 건너뛴다.

#### 대조 방법

1. PRD의 기능 요구사항(Functional Requirements) 섹션에서 항목 추출
2. 각 항목에 대해 git diff + 코드를 확인하여 상태 판정:

| 상태 | 의미 | 표기 |
|------|------|------|
| ✅ 구현 완료 | PRD 요구사항대로 구현됨 | 파일:행 참조 포함 |
| ❌ 미구현 | 구현되지 않음 | 사유 필수 기록 |
| △ 변경 구현 | PRD와 다르게 구현됨 | 변경 내용 + 변경 사유 |

#### 판단 기준

| 조건 | 판정 |
|------|------|
| ❌ 핵심 기능 미구현 | **Critical** → P4 재진입 필수 (BLOCKED) |
| ❌ 부가 기능 미구현 | **Important** → 사용자 판단 |
| △ 합리적 사유로 변경 | **Acceptable** → PRD 업데이트만 (Step C에서 처리) |
| 모든 항목 ✅ 또는 Acceptable △ | 정합성 **통과** |

### Step 3: 코드 품질 및 아키텍처 리뷰 (Full 모드)

**이 Step은 Large 작업(full 모드)에서만 실행됩니다. Medium 작업은 Step 3-minimal로 이동합니다.**

[references/code-review-prompt.md](references/code-review-prompt.md)의 6-area 프레임워크를 적용한다.

**코드 품질** (`.claude/rules/code-conventions.md` 기준):
- 타입 안전성 (strict 모드 준수, `any` 사용 여부)
- 에러 처리 패턴
- 네이밍 규칙 준수
- 매직 넘버, 중복 코드
- import 규칙 준수
- 프로젝트 코딩 컨벤션 일관성

**아키텍처/설계** (`.claude/rules/domain-boundaries.md` 기준):
- 도메인 경계 존중
- 공유 패키지 수정 시 영향 범위
- 의존성 방향 (단방향)
- 기존 패턴과의 일관성

**문서/표준**:
- 자명하지 않은 로직의 인라인 주석
- 공유 타입 적절한 위치 정의

### Step 3-minimal: 최소 검증 (Medium 작업 전용)

**이 Step은 Medium 작업(minimal 모드)에서만 실행됩니다. Large 작업은 Step 3(full)을 수행합니다.**

minimal 모드에서는 **Critical 이슈만 검증**하고, 나머지는 건너뜁니다.

#### 3-1. 타입 안전성 (Critical만)

`.claude/rules/code-conventions.md` 기준:

- [ ] **`any` 타입 사용 여부** — 타입 안전성 손실, 런타임 오류 가능성
- [ ] **strict 모드 위반** — 컴파일러 경고 무시
- [ ] **타입 단언 남용** — 타입 가드 우선 사용

#### 3-2. 도메인 경계 (Critical만)

`.claude/rules/domain-boundaries.md` 기준:

- [ ] **다른 도메인 폴더로의 import 여부** — 아키텍처 경계 위반
- [ ] **공유 패키지 오염 여부** — 도메인 로직이 공유 패키지에 포함됨
- [ ] **순환 의존성** — 도메인 간 양방향 참조

#### 3-3. 보안 (Critical만)

`.claude/rules/security.md` 기준:

- [ ] **시크릿 하드코딩** — API 키, 비밀번호, 토큰 노출
- [ ] **인증/인가 우회 가능성** — 인증 체크 누락
- [ ] **클라이언트에 민감 데이터 노출**

#### 3-4. 에러 처리 (Critical만)

`.claude/rules/code-conventions.md` 기준:

- [ ] **async 함수의 예외 처리 누락**
- [ ] **에러 경계 필요성** — 컴포넌트 트리 전체 중단 가능
- [ ] **사용자에게 기술적 에러 메시지 노출** — UX 저하

#### Step 3-minimal 출력 형식

**Critical 이슈 없을 때**:
```
# 코드 리뷰 보고서 (Minimal Mode)

## 검증 완료
- ✅ 타입 안전성: 이상 없음
- ✅ 도메인 경계: 준수
- ✅ 보안: 이상 없음
- ✅ 에러 처리: 적절

## Critical 이슈 (0건)

**판정**: ✅ 승인

---
**검증 모드**: minimal (Medium 작업)
```

**Critical 이슈 발견 시**:
```
# 코드 리뷰 보고서 (Minimal Mode)

## Critical 이슈 (N건)

### 🔴 Critical-1: [이슈 제목]
- **파일**: [경로:행]
- **내용**: [구체적 설명]
- **위험**: [영향]
- **수정안**: [수정 방법]

**판정**: ❌ P4 재진입 필요

---
**검증 모드**: minimal (Medium 작업)
**다음 단계**: Critical 이슈 수정 후 P5 Step A(minimal) 재실행
```

### Step 4: 이슈 분류

발견된 이슈를 심각도별로 분류한다.

| 심각도 | 기준 | 예시 |
|--------|------|------|
| **Critical** | 보안 취약점, 데이터 손실, 프로덕션 장애 | 인증 우회, 키 노출, 무한 루프 |
| **Important** | 기능 오동작, 성능 저하, 컨벤션 심각 위반 | 에러 처리 누락, N+1 쿼리, any 타입 |
| **Suggestion** | 코드 개선, 스타일, 선택적 최적화 | 네이밍 개선, 불필요한 re-render |

각 이슈에 반드시 포함: **파일:행**, **설명**, **영향**, **수정 권고**

### Step 5: 리뷰 보고서 생성

다음 형식으로 보고서를 생성한다:

```
코드 리뷰 보고서
================

리뷰 범위: [N]개 파일, [+추가/-삭제] 행
작업 유형: [신규 기능/확장/버그 수정/리팩토링]

### 강점
- [잘 구현된 부분, 파일:행 참조]

### Critical ([N]건)
1. **[이슈 제목]**
   - 파일: [경로:행]
   - 발견: [구체적 설명]
   - 영향: [영향 범위]
   - 수정 권고: [구체적 방법]

### Important ([N]건)
(동일 형식)

### Suggestion ([N]건)
1. **[이슈 제목]**
   - 파일: [경로:행]
   - 설명: [개선 방향]

### 권고사항
- [전체적 개선 방향]

### 종합 평가
**판정**: [승인 / 조건부 승인 / 수정 필요]
**사유**: [1~2문장 기술적 평가]
```
