SKILL.md에서 참조하는 서비스 기획 관련 프레임워크 및 템플릿입니다.

---

## 1. User Story 작성 가이드

### 형식
```
[역할]로서, [목표]하기 위해, [기능]하고 싶다.
```

### 좋은 User Story의 INVEST 기준

| 기준 | 의미 | 확인 방법 |
|------|------|----------|
| **I**ndependent | 다른 스토리와 독립적 | 순서 없이 개발 가능한가? |
| **N**egotiable | 협의 가능 | 구현 방식이 고정되어 있지 않은가? |
| **V**aluable | 가치 있음 | 사용자나 비즈니스에 가치를 제공하는가? |
| **E**stimable | 추정 가능 | 개발 비용을 추정할 수 있는가? |
| **S**mall | 적정 크기 | 1 스프린트 내 완료 가능한가? |
| **T**estable | 테스트 가능 | 완료 여부를 검증할 수 있는가? |

### 예시

**좋은 User Story:**
- "구매자로서, 장바구니에 담긴 상품의 합계를 확인하기 위해, 장바구니 페이지에서 총 금액을 보고 싶다."

**나쁜 User Story:**
- "시스템은 빠르게 동작해야 한다." → 비기능 요구사항이지 User Story가 아님
- "사용자로서, 모든 기능을 사용하고 싶다." → 너무 큼, INVEST의 S 위반

### 분해 계층

```
Theme (테마): "이커머스 플랫폼 구축"
  ↓
Epic (에픽): "상품 검색 시스템"
  ↓
Feature (기능): "키워드 검색"
  ↓
User Story: "구매자로서, 원하는 상품을 찾기 위해, 검색창에 키워드를 입력하고 싶다."
  ↓
Requirement (요구사항): "FR-001: 검색창에 2자 이상 입력 시 실시간 자동완성"
```

## 2. MoSCoW 우선순위 가이드

### 정의

| 우선순위 | 의미 | 비율 가이드 |
|---------|------|------------|
| **Must** | 없으면 서비스 론칭 불가. 핵심 가치를 전달하는 기능 | ~60% |
| **Should** | 중요하지만 Must 없이도 론칭은 가능. 사용자 경험 개선 | ~20% |
| **Could** | 있으면 좋지만 없어도 됨. 차별화 요소 | ~15% |
| **Won't** | 이번 버전에서는 하지 않음. 향후 고려 | ~5% |

### 판단 기준

| 질문 | Must | Should | Could |
|------|------|--------|-------|
| 이것 없이 서비스가 동작하는가? | 불가 | 가능하나 불편 | 가능 |
| 사용자가 즉시 필요로 하는가? | 즉시 | 곧 | 나중에 |
| 경쟁사 대비 차별화인가? | 기본 기능 | 경쟁 필수 | 차별화 |
| 기술적 의존성이 있는가? | 다른 기능의 전제 | 독립적 | 독립적 |

### 실전 팁
- **Must가 50%를 넘으면** → 범위를 재검토 (MVP를 더 좁혀야 할 수 있음)
- **Won't를 명시하는 것이 중요** → "이건 안 한다"를 분명히 해야 범위 크리프 방지
- 의심되면 **Should**로 분류 → Must 여부는 개발 진행 중 재평가

## 3. Acceptance Criteria 작성법

### Given-When-Then (GWT) 형식

```
Given: [사전 조건 - 시스템/사용자의 초기 상태]
When: [트리거 - 사용자가 수행하는 행동]
Then: [결과 - 예상되는 시스템 반응]
```

### 예시

**FR-001: 이메일 로그인**
```
Scenario 1: 정상 로그인
Given: 유효한 계정이 존재하는 사용자가 로그인 페이지에 있을 때
When: 올바른 이메일과 비밀번호를 입력하고 로그인 버튼을 클릭하면
Then: 대시보드 페이지로 이동하고, 사용자 이름이 헤더에 표시된다

Scenario 2: 잘못된 비밀번호
Given: 유효한 계정이 존재하는 사용자가 로그인 페이지에 있을 때
When: 올바른 이메일과 잘못된 비밀번호를 입력하고 로그인 버튼을 클릭하면
Then: "이메일 또는 비밀번호가 올바르지 않습니다" 에러 메시지가 표시된다

Scenario 3: 이메일 형식 검증
Given: 사용자가 로그인 페이지에 있을 때
When: 유효하지 않은 이메일 형식을 입력하면
Then: "올바른 이메일 형식을 입력해주세요" 인라인 에러가 표시된다
```

### 체크리스트 형식 (간략한 요구사항용)

```
- [ ] 이메일 형식 검증 (@ 포함, 도메인 확인)
- [ ] 비밀번호 8자 이상
- [ ] 로그인 실패 시 에러 메시지 표시
- [ ] 5회 연속 실패 시 계정 잠금 (30분)
- [ ] 로그인 성공 시 JWT 토큰 발급
```

### 어느 형식을 선택할 것인가

| 상황 | 권장 형식 |
|------|----------|
| 복잡한 분기/상태가 있는 기능 | GWT (시나리오별) |
| 단순한 CRUD 기능 | 체크리스트 |
| UI 상호작용이 핵심인 기능 | GWT |
| 데이터 처리/백엔드 로직 | 체크리스트 |

## 4. 기능 요구사항 테이블 템플릿

### 기능 요구사항 (FR)

| ID | Epic | Feature | Requirement | 우선순위 | 수용 기준 |
|----|------|---------|-------------|---------|----------|
| FR-001 | [Epic] | [Feature] | [구체적 요구사항] | Must | [GWT 또는 체크리스트] |
| FR-002 | [Epic] | [Feature] | [구체적 요구사항] | Should | [GWT 또는 체크리스트] |
| FR-003 | [Epic] | [Feature] | [구체적 요구사항] | Could | [GWT 또는 체크리스트] |

### ID 규칙
- `FR-xxx`: 기능 요구사항 (Functional Requirement)
- `NFR-xxx`: 비기능 요구사항 (Non-Functional Requirement)
- 번호는 001부터 순차적으로 부여
- Epic이 변경되어도 번호는 계속 증가 (중간 번호 재사용 안 함)

## 6. 정보 아키텍처 작성 가이드

### Site Map 작성 원칙

1. **깊이 3단계 이내** — 사용자가 3클릭 이내에 원하는 기능에 도달
2. **유사 기능 그룹핑** — 관련 기능은 같은 상위 메뉴 아래에 배치
3. **핵심 기능 우선** — 자주 사용하는 기능은 네비게이션 상단에 배치
4. **인증 상태 구분** — 로그인 전/후 접근 가능한 화면을 명확히 구분

### 네비게이션 패턴

| 패턴 | 적합한 서비스 | 특징 |
|------|-------------|------|
| 상단 GNB | 화면 수 적음 (5~7개) | 웹 표준, 익숙함 |
| 사이드바 | 화면 수 많음, 대시보드 | 확장성 좋음, 데스크톱 최적 |
| 하단 탭바 | 모바일 우선 | 엄지 접근성, 5개 이내 |
| 햄버거 메뉴 | 부가 기능 숨김 | 접근성 낮음, 보조용 |
