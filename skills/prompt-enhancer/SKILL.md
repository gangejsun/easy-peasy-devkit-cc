# Prompt Enhancer

사용자의 간략한 개발 요청을 프로젝트 컨텍스트를 반영하여 **강화된 프롬프트**로 변환합니다. 요청 규모에 따라 강화 수준이 달라집니다.

## Core Workflow

### Step 1: 프로젝트 컨텍스트 분석

**수집할 정보:**
- 프로젝트 구조 및 조직 (`CLAUDE.md`, 패키지 매니페스트, 설정 파일)
- 기술 스택 (프레임워크, 라이브러리 버전)
- 기존 패턴 (상태 관리, API 호출, 라우팅 방식)
- 코드 컨벤션 (`.claude/rules/code-conventions.md`)
- 유사한 기존 기능 (이미 구현된 것)

### Step 2: 요청 의도 및 규모 파악

사용자의 요청에서 다음을 식별:
- **작업 유형**: 신규 기능, 버그 수정, 리팩토링, API 연동, 기타
- **예상 규모**: Medium (1~2파일, ~50줄) vs Large (3+파일 또는 50줄+)
- **범위**: 단일 화면, 전체 플로우, 백엔드+프론트엔드
- **관련 의존성**: 연관된 기능이나 시스템

**강화 유형 판단:**

| 상황 | 강화 유형 | Step 3 동작 |
|------|----------|------------|
| P0 파이프라인 경유 (service-plan 존재) | P0-Tech | Step 3: P0 기술 명세 매핑 |
| 일반 요청 — Medium 규모 | Medium | Step 3: Medium 강화 |
| 일반 요청 — Large 규모 | Large | Step 3: Large 강화 |

**규모 판단 기준 (일반 요청 시):**
- **Medium**: 단순 변경. "버튼 색 바꿔줘", "버튼 추가해줘", "텍스트 수정", "스타일 조정" 등
- **Large**: 그 외 모든 기능 개발. 새 페이지, 새 API, 새 도메인, 상태 관리 추가 등
- 모호하면 **Large**로 진행

### Step 3: 규모별 강화 수행

---

#### P0 기술 명세 매핑 (P0-Tech)

service-planner의 기능 요구사항(FR-xxx)을 **프로젝트 코드 컨텍스트와 매핑**합니다.

**입력**: `dev/docs/service/service-plan-<name>.md`의 기능 요구사항 테이블

**프로세스**:
1. 프로젝트 코드베이스 분석 (Step 1과 동일)
2. 각 기능 요구사항(FR-xxx)에 대해:
   - **구현 위치**: 생성/수정할 파일 경로
   - **기존 패턴**: 참조할 유사 구현 (있으면)
   - **기술적 제약**: 프레임워크/라이브러리 제약
   - **의존성**: 관련 컴포넌트, 스토어, API
3. 비기능 요구사항(NFR-xxx)에 기술 스택 구현 방식 추가

**출력 형식**:
```markdown
# [기능명] 기술 명세 강화 요구사항

## 📋 프로젝트 컨텍스트
- Framework: [감지된 프레임워크 및 버전]
- Architecture: [감지된 아키텍처 패턴]
- State Management: [감지된 상태 관리]
- Key Libraries: [관련 라이브러리]

## 🔗 서비스 기획 참조
- 서비스 기획서: dev/docs/service/service-plan-<name>.md
- 사업 기획서: dev/docs/business/business-plan-<name>.md (해당 시)

## 📝 기술 명세 매핑된 기능 요구사항

### FR-001: [Feature명]
- **요구사항**: [service-plan의 원본 요구사항]
- **구현 위치**: [파일 경로]
- **기존 패턴**: [참조할 유사 구현]
- **기술적 구현**: [구체적 구현 방향]
- **수용 기준**: [service-plan의 원본 + 기술 수용 기준]

### FR-002: [Feature명]
...

## ✅ 통합 성공 기준
- [ ] [서비스 기획 수용 기준 + 기술 수용 기준 통합]
```

---

#### Medium 요청 (단순 변경, 1~2파일, ~50줄)

프로젝트 맥락을 반영한 **명확한 요청문**으로 강화합니다:

```markdown
# 강화된 요청

## 프로젝트 컨텍스트
- [감지된 기술 스택, 관련 패턴 요약 (2~3줄)]

## 강화된 요청문
[원본 요청을 프로젝트 맥락에 맞게 구체화한 1~3문장]

## 참고할 기존 패턴
- [유사 구현이나 관련 컨벤션 (있을 경우)]

## 확인 사항
- [모호한 부분 (1~3개)]
```

---

#### Large 요청 (3+파일 또는 50줄+)

프로젝트 컨텍스트 + 구현 범위 + **상세 요구사항과 성공 기준**까지 포함하여 강화합니다:

```markdown
# [기능명] 강화된 요청

## 📋 프로젝트 컨텍스트
- Framework: [감지된 프레임워크 및 버전]
- Architecture: [감지된 아키텍처 패턴]
- State Management: [감지된 상태 관리]
- Key Libraries: [관련 라이브러리]

## 🎯 구현 범위

### 주요 기능
1. [핵심 기능 1]
2. [핵심 기능 2]
3. [핵심 기능 3]

### 파일 구조
```
[프로젝트 패턴 기반 상세 파일 구조]
```

## 📝 상세 요구사항

### 1. [레이어/컴포넌트명]
- **위치**: [파일 경로]
- **목적**: [역할]
- **구현 내용**:
  - [구체적 요구사항 1]
  - [구체적 요구사항 2]
- **기존 패턴 따르기**: [참조할 기존 패턴]

### 2. [다음 레이어/컴포넌트]
...

## ✅ 성공 기준
- [ ] [수용 기준 1]
- [ ] [수용 기준 2]
- [ ] 기존 코드 스타일 및 아키텍처 일관성 유지

## 🔍 확인 사항
- [모호한 부분 및 결정 필요 항목]
```

---

### Step 4: 사용자 확인

강화된 프롬프트를 사용자에게 보여주고 확인:

```
프로젝트 컨텍스트를 분석하여 요청을 강화했습니다.

이 방향으로 진행해도 될까요?
수정하거나 추가할 내용이 있으면 말씀해주세요!
```

**Do NOT implement** until the user confirms. 목표는 다음 Phase에 전달할 명확한 요청을 만드는 것입니다.

## 예시

### Medium 예시

**사용자 입력:** `상품 카드에 참여자 수 배지 추가해줘`

**강화 출력:**
```markdown
# 강화된 요청

## 프로젝트 컨텍스트
- [감지된 프레임워크] + [감지된 스타일링 솔루션] + [감지된 UI 라이브러리]
- 기존 ProductCard 컴포넌트가 [감지된 경로]에 존재

## 강화된 요청문
ProductCard 컴포넌트에 현재 참여자 수를 보여주는 배지 UI 추가.
기존 카드 레이아웃의 우측 상단에 배치, participants count를 props로 전달.
프로젝트의 기존 UI 라이브러리 배지 컴포넌트 활용.

## 참고할 기존 패턴
- [프로젝트에서 사용 중인 배지/태그 컴포넌트 패턴]

## 확인 사항
- 최대 참여자 수 대비 현재 수 (예: "3/10")로 표시할까요, 현재 수만 표시할까요?
```

### Large 예시

**사용자 입력:** `로그인 기능 구현해줘`

**규모 판단**: Large (인증 전체 플로우, 6+ 파일 예상)

**강화 출력:**
```markdown
# 로그인 기능 강화된 요청

## 📋 프로젝트 컨텍스트
- Framework: [감지된 프레임워크 및 버전]
- State Management: [감지된 상태 관리 라이브러리]
- DB/Auth: [감지된 DB/인증 솔루션]
- Styling: [감지된 스타일링 솔루션]

## 🎯 구현 범위

### 주요 기능
1. 이메일/비밀번호 로그인 폼
2. [감지된 Auth 솔루션] 기반 인증
3. 로그인 성공 시 메인 페이지 이동
4. 에러 처리 및 사용자 피드백

### 파일 구조
[프로젝트 패턴 기반 파일 구조]
├── [auth 페이지 경로]              # 로그인 페이지
├── [컴포넌트 경로]/LoginForm.*     # 로그인 폼 컴포넌트
├── [lib 경로]/auth.*               # Auth 헬퍼
├── [상태관리 경로]/auth-store.*    # 인증 상태 스토어
└── [타입 경로]/auth.*              # 인증 관련 타입

## 📝 상세 요구사항

### 1. UI Layer - 로그인 페이지
- **위치**: [프로젝트 라우팅 패턴에 맞는 경로]
- **목적**: 사용자 로그인 UI 제공
- **구현 내용**:
  - 프로젝트의 컴포넌트 패턴에 맞게 구현
  - 기존 UI 라이브러리의 Input, Button 컴포넌트 사용
- **기존 패턴 따르기**: [감지된 컴포넌트 작성 패턴]

### 2. State Management
- **위치**: [프로젝트의 상태 관리 디렉토리]
- **구현 내용**: 인증 상태 스토어, user/loading/error 상태, login/logout 액션
- **기존 패턴 따르기**: [감지된 상태 관리 패턴]

### 3. Auth Layer
- **위치**: [프로젝트의 lib/서비스 디렉토리]
- **구현 내용**: 인증 API 연동, 세션 관리

## ✅ 성공 기준
- [ ] 이메일/비밀번호로 로그인 가능
- [ ] 인증 세션 생성
- [ ] 성공 시 메인 페이지 이동
- [ ] 실패 시 에러 메시지 표시
- [ ] 프로젝트 타입 시스템 준수

## 🔍 확인 사항
- 소셜 로그인(구글, 카카오 등)도 함께 구현할까요?
- "비밀번호 찾기" 기능이 필요한가요?
- 회원가입 페이지도 함께 만들까요?
```

## Tips

- 기존 화면/컴포넌트가 있으면 "기존 X와 유사한 접근으로" 형태로 참조
- 연관 기능(기존 Store, API 헬퍼 등)이 있으면 언급
- 프로젝트 컨텍스트가 불충분하면 사용자에게 추가 정보 요청
- 규모 판단이 애매하면 Large로 강화

## 프로젝트 커스텀 리소스

> 아래 경로에 `.md` 파일이 존재하면 프롬프트 강화 시 자동으로 참조됩니다. 파일이 없으면 이 섹션은 무시됩니다.

| 카테고리 | 경로 | 이 스킬에서의 용도 |
|---------|------|-------------------|
| 도메인 지식 | `.claude/resources/domain-knowledge/` | 요구사항 강화 시 비즈니스 규칙·도메인 용어를 프롬프트에 반영 |
