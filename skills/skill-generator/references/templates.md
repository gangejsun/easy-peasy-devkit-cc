## Description 작성 예시

### Pushy 수준 비교

| 수준 | 예시 | 판단 |
|------|------|------|
| 적정 | "사용자가 새로운 기능 개발을 요청했을 때, /dev/docs/prd 경로에 관련 PRD가 없으면 자동으로 PRD를 생성합니다. 기능 개발 전 PRD 검색 및 생성이 필요할 때 사용하세요. (project)" | 트리거 조건 명확, 상황 구체적 |
| 과도 | "모든 개발 요청에 대해 항상 PRD를 자동으로 생성합니다." | "모든", "항상" 무조건적 수식어 |
| 부족 | "PRD를 생성하는 스킬입니다." | 트리거 조건 없음 |

### Should-trigger / Should-not-trigger 작성법

스킬 검증 시 should-trigger 3개 + should-not-trigger 3개를 작성하여 description이 적절한지 판단.

**예시 (prd-generator)**:
- **Should-trigger**: "알림 기능을 개발해줘" → 신규 기능, PRD 필요
- **Should-trigger**: "결제 페이지에 카카오페이 추가해줘" → 기능 확장, PRD 확인 필요
- **Should-trigger**: "사용자 프로필 기능 추가" → 신규 기능
- **Should-not-trigger**: "로그인 버튼 색상 변경" → 단순 UI 수정, PRD 불필요
- **Should-not-trigger**: "README 업데이트" → 문서 작업
- **Should-not-trigger**: "타입 에러 수정해줘" → 버그 수정

핵심: should-not-trigger는 "명백히 무관한 것"이 아니라 **"경계선에 있지만 트리거되면 안 되는 것"**을 작성해야 실질적 검증이 됨.

## 예시: 코드 리뷰 스킬 생성

**입력:**
```
PR이 생성되면 자동으로 코드 리뷰를 수행하는 스킬을 만들어줘.
변경된 파일을 분석하고, 코드 품질/보안/성능 관점에서 피드백을 줘야 해.
```

**생성되는 스킬:**

```markdown
---
name: code-reviewer
description: PR 또는 코드 변경사항에 대해 자동으로 코드 리뷰를 수행합니다. 사용자가 코드 리뷰를 요청하거나 PR 피드백이 필요할 때 사용하세요. (project)
---

# Code Reviewer

코드 변경사항을 분석하여 품질, 보안, 성능 관점에서 구조화된 리뷰를 제공합니다.

## 워크플로우

### Step 1: 리뷰 대상 확인 및 분석 범위 결정

| 변경 유형 | Step 2 분석 범위 | Step 3 출력 |
|----------|-----------------|------------|
| 신규 기능 추가 | 코드 품질 + 보안 + 성능 전체 | 전체 리뷰 리포트 |
| 버그 수정 | 수정 로직 + 사이드 이펙트 중심 | 수정 검증 리포트 |
| 리팩토링 | 동작 보존 + 구조 개선 검증 | 변경 전후 비교 |

1. `git diff` 또는 `git log`으로 변경 파일 목록 확인
2. 변경된 파일의 내용을 분석

### Step 2: 관점별 분석
- **코드 품질**: 타입 안전성, 네이밍, 중복 코드
- **보안**: OWASP Top 10 취약점, 인증/인가
- **성능**: 불필요한 렌더링, N+1 쿼리, 메모리 누수

### Step 3: 리뷰 결과 정리
- 심각도별 분류 (Critical / Warning / Info)
- 파일별 인라인 코멘트
- 전체 요약

## 출력 형식

| 심각도 | 파일 | 라인 | 내용 |
|--------|------|------|------|
| Critical | [파일] | [라인] | [설명] |
| Warning | [파일] | [라인] | [설명] |
```
