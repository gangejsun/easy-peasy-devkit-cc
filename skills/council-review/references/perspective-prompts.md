# 관점별 분석 프롬프트 템플릿

Council Review의 3개 관점 Agent에게 전달할 프롬프트 템플릿입니다.
각 프롬프트에 `{context_brief}`를 치환하여 사용합니다.

---

## 기술 관점 프롬프트 (Technical Perspective)

```
# Technical Perspective Analysis

기술적 관점에서 제안된 기능을 분석합니다.

## 분석 대상
{context_brief}

## 분석 범위

### 1. 기술적 실현 가능성
- 현재 기술 스택(Next.js 15, TypeScript, Supabase, Tailwind CSS v4)으로 구현 가능한가?
- 기술적 불확실성이나 리스크는?
- 복잡도 추정: Low / Medium / High

### 2. 아키텍처 영향
- 기존 아키텍처에 어떻게 맞는가?
- 새로운 모듈/서비스 필요 여부
- 기존 코드 영향 (breaking changes, 마이그레이션 필요성)

### 3. 구현 접근법
- 권장하는 기술적 접근 방식 (간략한 근거 포함)
- 핵심 기술 결정 사항
- 외부 서비스/API 의존성

### 4. 성능 및 확장성
- 성능 영향 (DB 쿼리, API 호출, 렌더링)
- 현재 및 향후 규모에서의 확장성 우려

### 5. 범위 권고 (기술 렌즈)
- 기술적으로 필수인 기능
- 기술적으로 복잡하여 초기 가치 대비 지연 가능한 기능
- 기술 부채 고려 사항

### 6. 공수 추정
- 예상 수정 파일 수
- 복잡도: Low / Medium / High
- 공수 증가 리스크 요인

### 7. 블라인드스팟 (기술 관점에서 보이지 않는 것)
- 이 기능에 대해 아직 검증되지 않은 기술적 가정은?
- "당연하다"고 전제하고 있지만 실제로 확인이 필요한 것은?
- 비즈니스/UX 관점에서는 보이지만 기술 관점에서 간과할 수 있는 것은?

## 출력 형식
위 섹션별로 구조화하여 응답하세요.
마지막에 3줄 요약: "기술 관점 입장: [1문장 권고]"

상태: DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED
```

## 비즈니스 관점 프롬프트 (Business Perspective)

```
# Business Perspective Analysis

비즈니스 가치와 ROI 관점에서 제안된 기능을 분석합니다.

## 분석 대상
{context_brief}

## 분석 범위

### 1. 비즈니스 가치
- 이 기능이 해결하는 비즈니스 문제는?
- KPI에 미치는 예상 영향
- 매출/비용 영향

### 2. 시장 적합성
- 타겟 고객 니즈와 부합하는가?
- 경쟁 차별화 가능성
- 시장 타이밍 고려 사항

### 3. 우선순위 평가
- 긴급성: 시간에 민감한가?
- 영향도 vs 공수 매트릭스 위치
- 이 기능 대신 다른 것을 만들 때의 기회비용

### 4. 범위 권고 (비즈니스 렌즈)
- 비즈니스 가치 전달에 필수인 기능
- 가치를 높이지만 필수는 아닌 기능
- 비즈니스 영향이 낮아 지연 가능한 기능

### 5. 리스크 평가
- 비즈니스 리스크 (시장, 규제, 경쟁)
- 이 기능을 만들지 않았을 때의 비용
- 잘못 만들었을 때의 비용

### 6. 성공 기준
- 출시 후 성공을 어떻게 측정할 것인가
- 최소한의 성공 지표
- 가치 실현까지 예상 기간

### 7. 블라인드스팟 (비즈니스 관점에서 보이지 않는 것)
- 이 기능의 비즈니스 가치에 대해 아직 검증되지 않은 가정은?
- "고객이 원할 것"이라고 전제하고 있지만 실제로 확인이 필요한 것은?
- 기술/UX 관점에서는 보이지만 비즈니스 관점에서 간과할 수 있는 것은?

## 출력 형식
위 섹션별로 구조화하여 응답하세요.
마지막에 3줄 요약: "비즈니스 관점 입장: [1문장 권고]"

상태: DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED
```

## UX 관점 프롬프트 (User Experience Perspective)

```
# User Experience Perspective Analysis

사용자 경험과 사용성 관점에서 제안된 기능을 분석합니다.

## 분석 대상
{context_brief}

## 분석 범위

### 1. 사용자 니즈 검증
- 실제 사용자 Pain Point를 해결하는가?
- 사용자가 현재 이 문제를 어떻게 해결하고 있는가?
- 사용자 수요의 근거

### 2. 사용자 여정 영향
- 기존 사용자 흐름에 어떻게 통합되는가?
- 새로운 화면/인터랙션 필요 여부
- 인지 부하 평가

### 3. 사용성 우려
- 잠재적 사용성 함정
- 사용자를 혼란시킬 수 있는 엣지 케이스
- 접근성(A11y) 고려 사항

### 4. 범위 권고 (UX 렌즈)
- 좋은 사용자 경험에 필수인 기능
- UX를 개선하지만 나중에 추가 가능한 기능
- UX에 해로울 수 있는 기능

### 5. 인터랙션 설계 노트
- 사용할 핵심 인터랙션 패턴
- 에러 상태 및 빈 상태 고려 사항
- 점진적 공개(Progressive Disclosure) 기회

### 6. 사용자 리스크
- 사용자 혼란 또는 이탈 리스크
- 새 기능의 학습 곡선
- 기존 사용자 워크플로우에 미치는 영향

### 7. 블라인드스팟 (UX 관점에서 보이지 않는 것)
- 이 기능의 사용자 경험에 대해 아직 검증되지 않은 가정은?
- "사용자가 이렇게 행동할 것"이라고 전제하고 있지만 실제로 확인이 필요한 것은?
- 기술/비즈니스 관점에서는 보이지만 UX 관점에서 간과할 수 있는 것은?

## 출력 형식
위 섹션별로 구조화하여 응답하세요.
마지막에 3줄 요약: "UX 관점 입장: [1문장 권고]"

상태: DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED
```
