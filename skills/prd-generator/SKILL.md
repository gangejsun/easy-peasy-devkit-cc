# PRD Generator

신규 기능 개발 요청 시 관련 PRD(Product Requirements Document)가 존재하지 않을 경우 자동으로 PRD 문서를 생성하는 스킬입니다.

## 입력 처리

이 스킬은 다양한 방식으로 호출될 수 있습니다:

| 호출 상황 | 입력 | 동작 |
|----------|------|------|
| planning-agent에서 강화된 요구사항과 함께 호출 | 상세한 요구사항 텍스트 | 그것을 기반으로 PRD 생성 (Step 3 수집 생략) |
| 사용자가 `/prd-generator`로 직접 호출 | 간략한 요청 | Step 3에서 사용자에게 직접 수집 |
| 기타 | 다양한 수준의 정보 | 부족한 정보만 추가 수집 |

## 핵심 워크플로우

### Step 1: 기존 PRD 검색

`/dev/docs/prd/` 폴더에서 요청된 기능과 관련된 PRD를 검색합니다.

```bash
# PRD 폴더 탐색
ls /dev/docs/prd/

# 키워드로 관련 파일 검색
grep -l "[기능 키워드]" /dev/docs/prd/*.md
```

**검색 기준:**
- 파일명에 관련 키워드 포함 여부
- 파일 내용에 관련 기능 설명 존재 여부
- 최근 수정된 PRD 우선 확인

### Step 2: PRD 존재 여부 판단

| 상황 | 액션 |
|------|------|
| 관련 PRD 존재 | 사용자에게 기존 PRD 안내, 활용 여부 확인 |
| PRD 부재 | Step 3로 진행하여 신규 PRD 생성 |
| 부분적 PRD 존재 | 기존 PRD 보완 또는 신규 생성 선택 제안 |

### Step 3: 요구사항 수집

사용자로부터 다음 정보를 수집합니다:

**필수 정보:**
1. **기능명**: 기능의 명확한 이름
2. **목적**: 왜 이 기능이 필요한가?
3. **핵심 기능**: 주요 기능 목록
4. **타겟 사용자**: 누가 사용하는가?

**선택 정보:**
5. **기술적 제약**: 특별한 기술 요구사항
6. **우선순위**: 구현 우선순위
7. **마감일**: 예상 마감일

### Step 4: PRD 문서 생성

아래 템플릿을 기반으로 PRD를 생성합니다:

```markdown
# PRD: [기능명]

**문서 버전**: v1.0
**작성일**: [YYYY.MM.DD]
**상태**: Draft

---

## 1. 개요 (Overview)

### 1.1 기능명
[기능의 명확한 이름]

### 1.2 목적
[이 기능이 필요한 이유, 해결하려는 문제]

### 1.3 핵심 가치 제안
| 구분 | 내용 |
|-----|------|
| **문제** | [해결하려는 문제] |
| **솔루션** | [제안하는 해결책] |
| **차별점** | [기존 대비 차별화 포인트] |

---

## 2. 타겟 사용자

### 2.1 Primary Persona
| 항목 | 내용 |
|------|------|
| 직업/역할 | [사용자 직업] |
| 니즈 | [주요 니즈] |
| Pain Point | [해결하려는 불편함] |

### 2.2 사용자 시나리오
```
[사용자가 기능을 사용하는 시나리오 설명]
```

---

## 3. 기능 요구사항

### 3.1 핵심 기능 (Must Have)

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| F1 | [기능 설명] | P0 |
| F2 | [기능 설명] | P0 |

### 3.2 부가 기능 (Nice to Have)

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| F3 | [기능 설명] | P1 |

### 3.3 제외 범위 (Out of Scope)
- [이번 버전에서 제외되는 기능]

---

## 4. 기술 요구사항

### 4.1 시스템 아키텍처
[아키텍처 다이어그램 또는 설명]

### 4.2 기술 스택
| 영역 | 기술 | 선정 이유 |
|------|------|----------|
| Frontend | [기술] | [이유] |
| Backend | [기술] | [이유] |
| Database | [기술] | [이유] |

### 4.3 API 요구사항
[필요한 API 엔드포인트 목록]

### 4.4 데이터 모델
[필요한 데이터 스키마]

---

## 5. UI/UX 요구사항

### 5.1 주요 화면
[화면 목록 및 설명]

### 5.2 사용자 플로우
```
[Step 1] → [Step 2] → [Step 3] → [완료]
```

---

## 6. 성공 지표 (KPI)

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| [지표명] | [목표치] | [측정 방식] |

---

## 7. 일정 및 마일스톤

### 7.1 개발 단계
| 단계 | 기간 | 산출물 |
|------|------|--------|
| 설계 | [기간] | [산출물] |
| 개발 | [기간] | [산출물] |
| 테스트 | [기간] | [산출물] |

---

## 8. 리스크 및 대응

| 리스크 | 영향도 | 대응 방안 |
|--------|--------|----------|
| [리스크] | [높음/중간/낮음] | [대응책] |

---

## 부록

### 용어 정의
| 용어 | 정의 |
|------|------|
| [용어] | [설명] |

### 참고 자료
- [관련 문서 링크]
```

### Step 5: 파일 저장

생성된 PRD를 저장합니다:

- **저장 경로**: `/dev/docs/prd/`
- **파일명 규칙**: `prd-[feature-name].md` (kebab-case)
- **예시**: `prd-user-authentication.md`, `prd-notification-system.md`

### Step 6: 사용자 확인

생성된 PRD를 사용자에게 보여주고 다음을 확인합니다:

```
PRD 문서가 생성되었습니다.

📄 파일 위치: /dev/docs/prd/prd-[feature-name].md

검토 후 다음 사항을 확인해주세요:
1. 기능 범위가 적절한가요?
2. 추가하거나 수정할 내용이 있나요?
3. 이 PRD를 기반으로 개발을 진행해도 될까요?
```

## 기존 PRD 참조

PRD 생성 시 `/dev/docs/prd/` 폴더의 기존 PRD를 참조하여 일관성을 유지합니다:

**참조 포인트:**
- 문서 구조 및 섹션 구성
- 표 형식 및 다이어그램 스타일
- 기술 요구사항 작성 방식
- KPI 및 성공 지표 정의 방식

## 프로젝트 컨텍스트 활용

PRD 생성 시 다음 프로젝트 파일을 참조합니다:

| 파일 | 참조 목적 |
|------|----------|
| `CLAUDE.md` | 프로젝트 아키텍처, 기술 스택, 개발 가이드라인 |
| `dev/docs/` | 기존 아키텍처, API, 보안 문서 참조 |
| `.claude/rules/` | 프로젝트별 코딩/API/DB 규칙 참조 |

## 예시

### 입력
```
사용자: "사용자 알림 기능을 개발해줘. 이메일과 푸시 알림을 지원해야 해."
```

### 워크플로우 실행

1. `/dev/docs/prd/`에서 "알림", "notification" 키워드로 검색
2. 관련 PRD 없음 확인
3. 사용자에게 추가 정보 요청:
   ```
   알림 기능 PRD를 생성하겠습니다. 몇 가지 확인이 필요합니다:

   1. 어떤 이벤트에서 알림이 발송되어야 하나요?
   2. 알림 설정(수신 동의, 시간대 등)을 사용자가 관리할 수 있어야 하나요?
   3. 실시간 알림(WebSocket)도 필요한가요?
   ```
4. 정보 수집 후 PRD 생성
5. `/dev/docs/prd/prd-notification-system.md` 저장
6. 사용자 확인 요청

### 출력
```
✅ PRD가 생성되었습니다!

📄 /dev/docs/prd/prd-notification-system.md

## 생성된 PRD 요약

**기능명**: 사용자 알림 시스템
**핵심 기능**:
- 이메일 알림 발송
- 푸시 알림 발송
- 알림 설정 관리
- 알림 히스토리 조회

이 PRD를 검토하시고 수정이 필요하면 말씀해주세요.
PRD 확정 후 개발 문서(dev/active/)를 생성하겠습니다.
```

## 프로젝트 커스텀 리소스

> 아래 경로에 `.md` 파일이 존재하면 PRD 작성 시 자동으로 참조됩니다. 파일이 없으면 이 섹션은 무시됩니다.

| 카테고리 | 경로 | 이 스킬에서의 용도 |
|---------|------|-------------------|
| 도메인 지식 | `.claude/resources/domain-knowledge/` | PRD 작성 시 비즈니스 규칙, 도메인 용어집, 사용자 페르소나 반영 |
